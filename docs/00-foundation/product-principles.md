# Kodem Product Principles

Version: 1.0.0  
Status: Draft

---

# 1. Core Product Philosophy

## 1.1 Decision-First Product

Kodem does not exist to display data.

Kodem exists to help users make better business decisions.

Every feature MUST contribute to decision making, not reporting.

---

## 1.2 Action Over Information

Raw data is not valuable unless it leads to action.

Every screen SHOULD prioritize:

- What changed
- What matters
- What to do next

Over:

- Lists
- Tables
- Metrics without context

---

## 1.3 PLG First

Kodem is a Product-Led Growth system.

Users must be able to:

- Understand value without training
- Experience value within minutes
- Expand usage naturally over time

No feature should depend on sales assistance to be understood.

---

## 1.4 Time To First Value (TTFV)

New users MUST reach meaningful value within minutes of first login.

"Value" means:

- A useful insight
- A meaningful recommendation
- A clear understanding of their business state

Not:

- Setup completion
- Empty dashboards
- Configuration steps

---

## 1.5 Continuous Value Delivery

Value is not a one-time event.

Kodem MUST continuously deliver:

- Insights
- Recommendations
- Observations
- Improvements

Even if the user does nothing.

---

## 1.6 Progressive Complexity

The system MUST start simple and become more powerful over time.

Users should not be exposed to:

- Advanced configuration
- Complex data models
- Full system capabilities

Until they need them.

---

## 1.7 Invisible Complexity

Complexity belongs to the platform, not the user.

The system may include:

- Events
- Engines
- Models
- AI systems
- Rules

But the user should only see:

- Business state
- Insights
- Actions

---

## 1.8 Business Context First

Every feature MUST operate within a Business Context.

No feature should exist without understanding:

- What the business is
- How it operates
- What its goals are
- What changed recently

---

## 1.9 Explainability

Every insight or recommendation MUST be explainable.

The system MUST be able to answer:

- Why is this shown?
- What data was used?
- What triggered it?

No black-box recommendations in the UI.

---

## 1.10 Trust Through Transparency

Users must be able to trust the system.

Therefore:

- AI suggestions must be explainable
- Data sources must be visible
- System assumptions must be inspectable

---

## 1.11 Business Knowledge Accumulation

The system improves by accumulating understanding of the business over time.

Every interaction should contribute to:

- Better insights
- Better predictions
- Better recommendations

---

## 1.12 Integration-Driven Value

Integrations are not infrastructure.

Integrations are value generators.

Every integration MUST unlock:

- New insights
- New automation possibilities
- Better understanding of the business

---

## 1.13 No Empty Screens

Every screen MUST provide immediate value.

Empty states must include:

- Explanation
- First action
- Expected benefit

No passive UI states.

---

## 1.14 Unified Experience

Users should feel they are using one system, not multiple modules.

Modules exist for structure, not for fragmentation.

---

## 1.15 Outcome-Oriented Design

Every feature MUST answer at least one of:

- What should I do next?
- What changed?
- What is at risk?
- What is working well?

If it does not answer one of these → it should not exist.

---

## 1.16 Business Reality Mirror

Kodem should reflect the real state of the business as accurately as possible.

Not ideal state.

Not configured state.

Actual operational state.

---

## 1.17 Learning by Default

The system MUST learn from:

- User actions
- Integrations
- Events
- Outcomes

Without requiring manual configuration.

---

## 1.18 No Feature Without Value Path

Every feature MUST define:

- How it creates value
- How it improves business outcomes
- How it contributes to Business Knowledge

If this cannot be defined → feature is not valid.

---

## 1.19 Minimal Cognitive Load

Users should not need to think about:

- Data structure
- System architecture
- Integration logic

Only:

- Their business
- Their decisions
- Their next actions

---

## 1.20 AI as Amplifier, Not Source

AI is used to:

- Interpret
- Summarize
- Recommend
- Predict

AI is NOT the source of truth.

All AI output MUST be grounded in real system data.
